## Introduction to RAG

### What is RAG?

**Retrieval-Augmented Generation (RAG)** is a technique that combines information retrieval with text generation to produce more accurate, factual, and up-to-date responses.

```
Traditional LLM:
User Query → LLM → Answer (based only on training data)

RAG:
User Query → Retrieve Relevant Docs → LLM + Docs → Answer
```

### Why RAG?

**Problems with Pure LLMs**:
1. **Knowledge Cutoff**: Can't access information after training
2. **Hallucinations**: May generate plausible but incorrect information
3. **Domain-Specific Knowledge**: Limited specialized knowledge
4. **No Source Attribution**: Can't cite sources

**RAG Solutions**:
```
Problem: "What happened in today's news?"
LLM alone: Cannot answer (knowledge cutoff)
RAG: Retrieves today's articles → Generates answer

Problem: "What is our company's vacation policy?"
LLM alone: Guesses or hallucinates
RAG: Retrieves company handbook → Accurate answer

Problem: "Summarize this 500-page document"
LLM alone: Limited context window
RAG: Retrieves relevant sections → Generates summary
```

### RAG vs Fine-tuning

| Aspect | RAG | Fine-tuning |
|--------|-----|-------------|
| **Knowledge Update** | Real-time (update documents) | Requires retraining |
| **Cost** | Low (add/update docs) | High (retrain model) |
| **Accuracy** | High (retrieves exact facts) | Depends on training data |
| **Latency** | Higher (retrieval step) | Lower (no retrieval) |
| **Source Attribution** | Easy (can cite sources) | Difficult |
| **Best For** | Factual QA, docs | Task adaptation, style |

**Example Use Cases**:
```
RAG:
- Customer support with knowledge base
- Legal document QA
- Medical diagnosis from papers
- Company internal search

Fine-tuning:
- Specific writing style
- Domain terminology
- Task-specific behavior
- Brand voice
```

## RAG Architecture

### Basic RAG Pipeline

```
┌─────────────────────────────────────────────────────┐
│               1. INDEXING (Offline)                 │
├─────────────────────────────────────────────────────┤
│  Documents                                          │
│      ↓                                              │
│  Chunking (Split into smaller pieces)              │
│      ↓                                              │
│  Embedding (Convert to vectors)                    │
│      ↓                                              │
│  Store in Vector Database                          │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│               2. RETRIEVAL (Online)                  │
├─────────────────────────────────────────────────────┤
│  User Query                                         │
│      ↓                                              │
│  Embed Query                                        │
│      ↓                                              │
│  Search Vector Database (similarity)               │
│      ↓                                              │
│  Retrieve Top-K relevant chunks                    │
└─────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────┐
│               3. GENERATION (Online)                 │
├─────────────────────────────────────────────────────┤
│  Query + Retrieved Documents                        │
│      ↓                                              │
│  Build Prompt (Context + Question)                 │
│      ↓                                              │
│  LLM Generation                                     │
│      ↓                                              │
│  Final Answer                                       │
└─────────────────────────────────────────────────────┘
```

### Detailed Flow Example

```python
# Example: Customer Support RAG

# 1. INDEXING (done once)
documents = [
    "Our return policy allows 30 days...",
    "Shipping takes 3-5 business days...",
    "Premium members get free shipping..."
]

chunks = chunk_documents(documents)
embeddings = embed_model.encode(chunks)
vector_db.insert(embeddings, chunks)

# 2. USER QUERY
query = "What is the return policy?"

# 3. RETRIEVAL
query_embedding = embed_model.encode(query)
relevant_chunks = vector_db.search(query_embedding, top_k=3)

# Results:
# 1. "Our return policy allows 30 days..." (score: 0.92)
# 2. "Returns must include original packaging..." (score: 0.85)
# 3. "Refunds processed within 7 days..." (score: 0.78)

# 4. GENERATION
context = "\n".join([chunk.text for chunk in relevant_chunks])
prompt = f"""Answer the question based on the context below.

Context:
{context}

Question: {query}

Answer:"""

answer = llm.generate(prompt)
# "Our return policy allows returns within 30 days of purchase..."
```

## Document Processing and Chunking

### Why Chunking?

Documents are often too long for:
1. Embedding models (max 512 tokens for many models)
2. LLM context windows (need only relevant parts)
3. Precise retrieval (whole document may be less relevant than specific section)

```
Without Chunking:
Document (10,000 words) → Single embedding
Problem: Loses granularity, can't pinpoint relevant section

With Chunking:
Document → 50 chunks (200 words each) → 50 embeddings
Benefit: Can retrieve specific relevant sections
```

### Chunking Strategies

#### 1. Fixed Size Chunking

```python
# Split by character count
chunk_size = 500  # characters
overlap = 50      # overlap between chunks

text = "Long document text..."
chunks = []

for i in range(0, len(text), chunk_size - overlap):
    chunk = text[i:i + chunk_size]
    chunks.append(chunk)

Example:
Chunk 1: "The transformer architecture was introduced..." (chars 0-500)
Chunk 2: "...was introduced in 2017. It revolutionized..." (chars 450-950)
          └─ overlap ─┘
```

**Pros**: Simple, predictable size
**Cons**: May split sentences/paragraphs unnaturally

#### 2. Sentence-Based Chunking

```python
# Split by sentences, combine to target size
sentences = split_into_sentences(text)
chunks = []
current_chunk = []
current_size = 0

for sentence in sentences:
    if current_size + len(sentence) > max_chunk_size:
        chunks.append(" ".join(current_chunk))
        current_chunk = [sentence]
        current_size = len(sentence)
    else:
        current_chunk.append(sentence)
        current_size += len(sentence)

Example:
Chunk 1: "Transformers use attention. This allows parallel processing.
          The architecture has encoders and decoders."
Chunk 2: "Self-attention computes relationships. It uses query, key,
          and value matrices."
```

**Pros**: Preserves sentence boundaries
**Cons**: Variable chunk sizes

#### 3. Semantic Chunking

```python
# Split based on topic changes
# Use embedding similarity to detect topic shifts

embeddings = [embed(sent) for sent in sentences]
chunks = []
current_chunk = [sentences[0]]

for i in range(1, len(sentences)):
    similarity = cosine_sim(embeddings[i-1], embeddings[i])

    if similarity < threshold:  # Topic change detected
        chunks.append(" ".join(current_chunk))
        current_chunk = [sentences[i]]
    else:
        current_chunk.append(sentences[i])

Example:
Chunk 1: "Paris is the capital of France. It has 2.2M people.
          The Eiffel Tower is there."
          (All about Paris)

Chunk 2: "Machine learning uses data. Neural networks are popular.
          GPT is a language model."
          (New topic: ML - low similarity to Paris)
```

**Pros**: Maintains semantic coherence
**Cons**: Computationally expensive

#### 4. Document Structure-Based

```python
# Split by markdown headers, sections, etc.

def chunk_by_structure(markdown_text):
    sections = split_by_headers(markdown_text)
    chunks = []

    for section in sections:
        if len(section) > max_size:
            # Further split large sections
            sub_chunks = split_by_paragraph(section)
            chunks.extend(sub_chunks)
        else:
            chunks.append(section)

    return chunks

Example:
# Chapter 1: Introduction
Chunk 1: [All content under "Introduction"]

# Chapter 2: Methods
Chunk 2: [All content under "Methods"]
```

**Pros**: Preserves logical structure
**Cons**: Requires structured documents

### Chunk Size Considerations

```
Small Chunks (100-200 tokens):
Pros:
- Precise retrieval
- More specific context
Cons:
- May miss broader context
- More chunks to manage

Large Chunks (500-1000 tokens):
Pros:
- More context per chunk
- Fewer embeddings
Cons:
- Less precise
- May include irrelevant info

Recommended: 300-500 tokens with 50-100 token overlap
```

## Retrieval Strategies

### 1. Dense Retrieval (Vector Search)

```python
# Convert query and documents to embeddings
query_emb = embedding_model.encode(query)
doc_embs = [embedding_model.encode(doc) for doc in documents]

# Compute similarity
similarities = [cosine_sim(query_emb, doc_emb) for doc_emb in doc_embs]

# Retrieve top-k
top_k_indices = argsort(similarities)[-k:]

Example:
Query: "How does attention work?"
Top results:
1. "Attention mechanism computes..." (0.91)
2. "Self-attention uses Q, K, V..." (0.87)
3. "Multi-head attention splits..." (0.82)
```

### 2. Sparse Retrieval (BM25)

```python
# Traditional keyword-based search
# BM25 scoring based on term frequency

from rank_bm25 import BM25Okapi

corpus = ["doc1 text", "doc2 text", "doc3 text"]
tokenized_corpus = [doc.split() for doc in corpus]
bm25 = BM25Okapi(tokenized_corpus)

query = "attention mechanism"
scores = bm25.get_scores(query.split())

Example:
Query: "transformer attention"
- Scores highly if document contains exact words
- Good for keyword matching
- Complements dense retrieval
```

### 3. Hybrid Retrieval

```python
# Combine dense and sparse retrieval

def hybrid_search(query, alpha=0.5):
    # Dense retrieval
    dense_scores = vector_db.search(query)

    # Sparse retrieval
    sparse_scores = bm25.search(query)

    # Combine scores
    final_scores = {}
    for doc_id in all_docs:
        final_scores[doc_id] = (
            alpha * dense_scores[doc_id] +
            (1 - alpha) * sparse_scores[doc_id]
        )

    return top_k(final_scores)

Benefits:
- Dense: Captures semantic similarity
- Sparse: Captures exact keyword matches
- Hybrid: Best of both worlds
```

### 4. Re-ranking

```python
# Two-stage retrieval

# Stage 1: Fast retrieval (get top 100)
candidates = vector_db.search(query, top_k=100)

# Stage 2: Re-rank with cross-encoder (slower but more accurate)
cross_encoder = CrossEncoder('model-name')
pairs = [[query, doc] for doc in candidates]
scores = cross_encoder.predict(pairs)

# Get top-k after re-ranking
top_k = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)[:10]

Example:
Initial retrieval: 100 documents (fast bi-encoder)
Re-ranking: Score query-document pairs (slow but accurate cross-encoder)
Final: Top 10 most relevant documents
```

### 5. Query Expansion

```python
# Expand query with related terms

original_query = "GPU memory"

# Method 1: Synonyms
expanded = "GPU memory RAM VRAM video memory"

# Method 2: LLM expansion
expansion_prompt = f"Generate 3 alternative phrasings: {original_query}"
alternatives = llm.generate(expansion_prompt)
# ["graphics processing unit memory", "video card RAM", "GPU VRAM capacity"]

# Search with all variants
results = []
for variant in alternatives:
    results.extend(vector_db.search(variant))

# Deduplicate and rank
final_results = deduplicate_and_rank(results)
```

```python
def create_rag_prompt(query, retrieved_docs):
```

```python
def create_rag_prompt_with_sources(query, retrieved_docs):

Generated:
```

### Context Window Management

```python
# Ensure context fits in LLM's window

max_context_tokens = 3000  # Leave room for query and answer
retrieved_docs = vector_db.search(query, top_k=20)

# Truncate or select documents to fit
context = ""
token_count = 0

for doc in retrieved_docs:
    doc_tokens = count_tokens(doc.text)
    if token_count + doc_tokens <= max_context_tokens:
        context += doc.text + "\n\n"
        token_count += doc_tokens
    else:
        break

# Now context fits within limit
```

## Advanced RAG Techniques

### 1. HyDE (Hypothetical Document Embeddings)

```python
# Instead of embedding query directly, generate hypothetical answer first

query = "How does attention mechanism work?"

# Generate hypothetical answer
hyde_prompt = f"Write a detailed answer to: {query}"
hypothetical_answer = llm.generate(hyde_prompt)

# Embed hypothetical answer (not original query)
query_embedding = embed_model.encode(hypothetical_answer)

# Retrieve using hypothetical answer embedding
results = vector_db.search(query_embedding)

Benefit: Hypothetical answer is closer to actual documents than query
```

### 2. Multi-Query Retrieval

```python
# Generate multiple query variations

original_query = "transformer architecture"

variations_prompt = f"""Generate 3 different ways to ask about: {original_query}
1.
2.
3."""

variations = llm.generate(variations_prompt)
# ["how do transformers work",
#  "explain transformer neural networks",
#  "transformer model architecture"]

# Retrieve for each variation
all_results = []
for variant in variations:
    results = vector_db.search(variant, top_k=5)
    all_results.extend(results)

# Deduplicate and rank
final_results = reciprocal_rank_fusion(all_results)
```

### 3. Self-Query Retrieval

```python
# LLM extracts metadata filters from query

query = "Show me recent papers about transformers from 2023"

extraction_prompt = f"""Extract search parameters:
Query: {query}
Output JSON with: search_query, filters"""

extracted = llm.generate(extraction_prompt)
# {
#   "search_query": "papers about transformers",
#   "filters": {"year": 2023, "type": "paper", "date": "recent"}
# }

# Use extracted info for retrieval
results = vector_db.search(
    query=extracted["search_query"],
    filter=extracted["filters"]
)
```

### 4. Iterative Retrieval

```python
# Retrieve, generate, retrieve again if needed

query = "Complex multi-hop question"

# First retrieval
docs1 = vector_db.search(query, top_k=5)

# Initial answer
initial_answer = llm.generate(create_prompt(query, docs1))

# Check if more info needed
if needs_more_info(initial_answer):
    # Generate follow-up query
    followup = extract_followup_query(initial_answer)

    # Second retrieval
    docs2 = vector_db.search(followup, top_k=5)

    # Final answer
    all_docs = docs1 + docs2
    final_answer = llm.generate(create_prompt(query, all_docs))
```

### 5. Parent-Child Chunking

```python
# Store both small (child) and large (parent) chunks

# Child chunks for retrieval (precise)
child_chunks = split_text(document, chunk_size=200)

# Parent chunks for context (comprehensive)
parent_chunks = split_text(document, chunk_size=1000)

# Link children to parents
for child in child_chunks:
    child.parent_id = get_parent_chunk(child)

# Retrieve children, return parents
retrieved_children = vector_db.search(query, top_k=5)
parent_chunks_to_use = [get_parent(child) for child in retrieved_children]

# Generate with full parent context
answer = llm.generate(create_prompt(query, parent_chunks_to_use))
```

## RAG Implementation

### Complete Example

```python
import openai
from sentence_transformers import SentenceTransformer
import chromadb

# 1. Initialize components
embedding_model = SentenceTransformer('all-MiniLM-L6-v2')
chroma_client = chromadb.Client()
collection = chroma_client.create_collection("my_docs")

# 2. Index documents
documents = [
    "Transformers use attention mechanisms for sequence processing.",
    "GPT is a decoder-only transformer architecture.",
    "BERT uses bidirectional encoding for language understanding."
]

for i, doc in enumerate(documents):
    collection.add(
        ids=[f"doc_{i}"],
        documents=[doc],
        embeddings=[embedding_model.encode(doc).tolist()]
    )

# 3. RAG Query function
def rag_query(question, top_k=3):
    # Embed query
    query_embedding = embedding_model.encode(question)

    # Retrieve
    results = collection.query(
        query_embeddings=[query_embedding.tolist()],
        n_results=top_k
    )

    retrieved_docs = results['documents'][0]

    # Create prompt
    context = "\n".join(retrieved_docs)
    prompt = f"""Answer based on context:

Context:
{context}

Question: {question}

Answer:"""

    # Generate
    response = openai.ChatCompletion.create(
        model="gpt-3.5-turbo",
        messages=[{"role": "user", "content": prompt}]
    )

    return response.choices[0].message.content

# 4. Use it
answer = rag_query("What is GPT?")
print(answer)
# "GPT is a decoder-only transformer architecture."
```

### Production-Ready RAG

```python
class RAGSystem:
    def __init__(self, vector_db, embedding_model, llm):
        self.vector_db = vector_db
        self.embedding_model = embedding_model
        self.llm = llm

    def index_documents(self, documents, batch_size=100):
        """Index documents with batching"""
        for i in range(0, len(documents), batch_size):
            batch = documents[i:i+batch_size]
            embeddings = self.embedding_model.encode(batch)
            self.vector_db.insert(embeddings, batch)

    def retrieve(self, query, top_k=5, filters=None):
        """Retrieve relevant documents"""
        query_emb = self.embedding_model.encode(query)
        results = self.vector_db.search(
            query_emb,
            top_k=top_k,
            filters=filters
        )
        return results

    def generate(self, query, retrieved_docs):
        """Generate answer from retrieved context"""
        context = self._build_context(retrieved_docs)
        prompt = self._build_prompt(query, context)
        answer = self.llm.generate(prompt)
        return answer

    def query(self, question, top_k=5):
        """End-to-end RAG pipeline"""
        # Retrieve
        docs = self.retrieve(question, top_k)

        # Generate
        answer = self.generate(question, docs)

        # Return with sources
        return {
            "answer": answer,
            "sources": [doc.metadata for doc in docs]
        }
```

## Evaluation and Optimization

### RAG Evaluation Metrics

#### 1. Retrieval Metrics

```python
# Precision@K
def precision_at_k(retrieved, relevant, k):
    retrieved_k = retrieved[:k]
    return len(set(retrieved_k) & set(relevant)) / k

# Recall@K
def recall_at_k(retrieved, relevant, k):
    retrieved_k = retrieved[:k]
    return len(set(retrieved_k) & set(relevant)) / len(relevant)

# MRR (Mean Reciprocal Rank)
def mrr(retrieved, relevant):
    for i, doc in enumerate(retrieved):
        if doc in relevant:
            return 1 / (i + 1)
    return 0

Example:
Retrieved: [doc1, doc3, doc5, doc7, doc2]
Relevant: [doc2, doc3, doc6]

Precision@3: 1/3 (only doc3 in top-3)
Recall@5: 2/3 (doc3 and doc2 found)
MRR: 1/2 (first relevant doc at position 2)
```

#### 2. Generation Metrics

```python
# BLEU Score (n-gram overlap)
from nltk.translate.bleu_score import sentence_bleu

reference = "GPT is a language model"
generated = "GPT is a large language model"
score = sentence_bleu([reference.split()], generated.split())

# ROUGE Score (recall-oriented)
from rouge import Rouge

rouge = Rouge()
scores = rouge.get_scores(generated, reference)

# Semantic Similarity
from sentence_transformers import util

emb1 = model.encode(reference)
emb2 = model.encode(generated)
similarity = util.cos_sim(emb1, emb2)
```

#### 3. Faithfulness

```python
# Check if answer is grounded in context

def check_faithfulness(answer, context):
    prompt = f"""Does the answer contain information not in the context?

Context: {context}
Answer: {answer}

Output: Yes/No"""

    result = llm.generate(prompt)
    return result.lower() == "no"
```

## Key Takeaways

### Core Concepts

1. **RAG combines retrieval and generation** to produce accurate, grounded answers

2. **Three main stages**: Indexing (offline), Retrieval (online), Generation (online)

3. **Chunking is critical**: 300-500 tokens with 50-100 overlap recommended

4. **Vector similarity** enables semantic search beyond keyword matching

5. **Hybrid retrieval** (dense + sparse) often outperforms either alone

### Implementation

6. **Metadata enhances retrieval** (filtering, source attribution)

7. **Re-ranking improves quality** at cost of latency

8. **Prompt design matters**: Include context, citations, instructions

9. **Context window limits** require smart document selection

10. **Evaluation covers both** retrieval quality and generation quality

### Advanced Techniques

11. **HyDE** embeds hypothetical answers for better retrieval

12. **Multi-query** uses query variations for broader coverage

13. **Iterative retrieval** enables multi-hop reasoning

14. **Parent-child chunking** balances precise retrieval with complete context

### Optimization

15. **Trade-offs**: Chunk size, top-k, context length affect accuracy/latency
