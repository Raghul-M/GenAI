## Introduction to LLM Deployment

### Deployment vs Training

| Aspect | Training | Inference |
|--------|----------|-----------|
| **Frequency** | Once or periodic | Continuous |
| **Latency** | Hours/days OK | Milliseconds-seconds |
| **Throughput** | Lower | Higher |
| **Cost Focus** | One-time | Ongoing |
| **Optimization** | Training speed | Response time |

### Key Challenges

```
1. Latency
   - Large models slow to run
   - User expects <1s response
   - Challenge: Balance quality vs speed

2. Throughput
   - Many concurrent users
   - Each request uses GPU
   - Challenge: Serve 1000s requests/sec

3. Cost
   - GPU expensive ($1-3/hour)
   - Scales with users
   - Challenge: Keep costs reasonable

4. Model Size
   - 7B model = 14GB (FP16)
   - 70B model = 140GB
   - Challenge: Fit in GPU memory
```

### Deployment Options

```
1. Cloud API Services
   - OpenAI, Anthropic, Cohere
   - Pros: Easy, scalable, maintained
   - Cons: Cost, data privacy, dependency

2. Self-Hosted Cloud
   - AWS SageMaker, Google Vertex AI
   - Pros: Control, customization
   - Cons: Setup complexity, maintenance

3. On-Premise
   - Your own servers
   - Pros: Full control, data security
   - Cons: Hardware cost, expertise needed

4. Edge Deployment
   - On device (mobile, IoT)
   - Pros: Privacy, low latency
   - Cons: Limited resources, small models only
```

## Inference Optimization

### 1. Quantization

**Reduce model precision to use less memory and compute**

```
Precision Levels:
FP32 (32-bit): 4 bytes/param (training default)
FP16 (16-bit): 2 bytes/param (inference default)
INT8 (8-bit): 1 byte/param (quantized)
INT4 (4-bit): 0.5 bytes/param (aggressive)

Model Size:
7B parameters:
FP32: 28 GB
FP16: 14 GB
INT8: 7 GB
INT4: 3.5 GB
```

**Implementation**:
```python
from transformers import AutoModelForCausalLM, BitsAndBytesConfig

# 8-bit quantization
quantization_config = BitsAndBytesConfig(load_in_8bit=True)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b",
    quantization_config=quantization_config
)

# Inference
# ~50% memory reduction, ~10-20% slower
```

**Trade-offs**:
```
FP16 vs INT8:
Memory: 2x reduction
Speed: 1.5-2x faster
Quality: 1-2% drop (minimal)

FP16 vs INT4:
Memory: 4x reduction
Speed: 2-3x faster
Quality: 5-10% drop (noticeable)
```

### 2. KV Cache Optimization

**Cache key-value pairs to avoid recomputation**

```
Without KV Cache:
"The cat sat on the mat. The dog"
                          ↓
Recompute attention for entire sequence

With KV Cache:
"The cat sat on the mat. The dog"
└──────────cached────────┘  ↓
Only compute for "dog"

Speedup: 3-5x for generation
```

**Implementation**:
```python
# Automatically used by most frameworks
outputs = model.generate(
    input_ids,
    max_length=100,
    use_cache=True  # Default
)
```

**Memory Cost**:
```
KV Cache Size ≈ 2 × num_layers × hidden_size × sequence_length

Example (Llama-2-7B):
Layers: 32
Hidden: 4096
Sequence: 2048

KV Cache: 2 × 32 × 4096 × 2048 × 2 bytes
         ≈ 1 GB per request

Challenge: Large for long contexts
```

### 3. Batching

**Process multiple requests simultaneously**

```
Sequential (No Batching):
Request 1: 100ms
Request 2: 100ms
Request 3: 100ms
Total: 300ms

Batched (Batch size=3):
Requests 1,2,3 together: 120ms
Total: 120ms

Throughput: 3x higher
Latency per request: Similar
```

**Dynamic Batching**:
```python
# Wait for requests to accumulate
max_wait_time = 10ms  # Don't wait too long
max_batch_size = 32

while True:
    batch = []
    start_time = now()

    while len(batch) < max_batch_size:
        if now() - start_time > max_wait_time:
            break
        if request_available():
            batch.append(get_request())

    if batch:
        process_batch(batch)
```

### 4. Model Parallelism

**Split model across multiple GPUs**

```
Tensor Parallelism:
GPU 1: First half of each layer
GPU 2: Second half of each layer

Pipeline Parallelism:
GPU 1: Layers 1-16
GPU 2: Layers 17-32

Example (70B model):
Single A100 (80GB): Doesn't fit
2x A100: Fits with tensor parallelism
4x A100: Comfortable, faster
```

### 5. Flash Attention

**Optimized attention computation**

```
Standard Attention:
- Materializes full attention matrix
- Memory: O(N²)
- Time: O(N²)

Flash Attention:
- Fused operations
- Memory: O(N)
- Time: O(N²) but faster constants

Speedup: 2-4x on long sequences
Memory: 10-20x reduction
```

**Implementation**:
```python
from transformers import LlamaForCausalLM

model = LlamaForCausalLM.from_pretrained(
    "meta-llama/Llama-2-7b",
    attn_implementation="flash_attention_2"
)

# Automatically uses Flash Attention
```

### 6. Speculative Decoding

**Use small model to draft, large model to verify**

```
Standard Decoding (70B model):
Token 1: 50ms
Token 2: 50ms
Token 3: 50ms
Total: 150ms for 3 tokens

Speculative Decoding:
Draft (7B): 3 tokens in 20ms
Verify (70B): All 3 in 50ms
Total: 70ms for 3 tokens

Speedup: 2-3x
```

## Model Serving Frameworks

### 1. vLLM

**High-performance LLM inference server**

```python
from vllm import LLM, SamplingParams

# Load model
llm = LLM(model="meta-llama/Llama-2-7b")

# Configure sampling
sampling_params = SamplingParams(
    temperature=0.7,
    top_p=0.9,
    max_tokens=100
)

# Generate
prompts = ["Tell me about AI", "Explain Python"]
outputs = llm.generate(prompts, sampling_params)

# Features:
# - PagedAttention (efficient KV cache)
# - Continuous batching
# - Optimized CUDA kernels
# Throughput: 10-20x vs Hugging Face
```

### 2. Text Generation Inference (TGI)

**Hugging Face's inference server**

```bash
# Launch server
docker run -p 8080:80 \
  --gpus all \
  ghcr.io/huggingface/text-generation-inference:latest \
  --model-id meta-llama/Llama-2-7b

# Query
curl http://localhost:8080/generate \
  -X POST \
  -H 'Content-Type: application/json' \
  -d '{"inputs":"What is AI?","parameters":{"max_new_tokens":100}}'

# Features:
# - Token streaming
# - Dynamic batching
# - Quantization support
# - Easy deployment
```

### 3. NVIDIA Triton

**Multi-framework serving platform**

```python
# model_repository/llama/config.pbtxt
name: "llama"
backend: "python"
max_batch_size: 32

# Supports:
# - PyTorch, TensorFlow, ONNX
# - Dynamic batching
# - Model ensemble
# - Multi-GPU
# Production-grade
```

### 4. Ray Serve

**Scalable ML serving**

```python
from ray import serve
import torch

@serve.deployment(
    ray_actor_options={"num_gpus": 1},
    max_concurrent_queries=10
)
class LLMModel:
    def __init__(self):
        self.model = load_model("llama-7b")

    def __call__(self, request):
        return self.model.generate(request["prompt"])

serve.run(LLMModel.bind())

# Features:
# - Auto-scaling
# - Multi-model deployment
# - Resource management
```

### 5. Ollama (Local Development)

```bash
# Run locally
ollama run llama2

curl http://localhost:11434/api/generate -d '{
  "model": "llama2",
  "prompt": "Why is the sky blue?"
```

## Deployment Architectures

### 1. API Gateway Pattern

```
┌─────────┐
│ Client  │
└────┬────┘
     ↓
┌────────────┐
│ API Gateway│ (Load balancing, auth, rate limiting)
└─────┬──────┘
      ↓
┌─────────────────────────┐
│  Model Servers (Pool)   │
│  ┌──────┐  ┌──────┐    │
│  │ GPU1 │  │ GPU2 │ ...│
│  └──────┘  └──────┘    │
└─────────────────────────┘
```

### 2. Microservices Architecture

```
┌─────────┐
│ Client  │
└────┬────┘
     ↓
┌──────────────┐
│ API Service  │ (REST/GraphQL)
└──────┬───────┘
       ↓
┌─────────────────────────────┐
│ Inference Service (LLM)     │
└──────┬──────────────────────┘
       ↓
┌────────────────────┐
│ Vector DB Service  │ (for RAG)
└────────────────────┘
```

### 3. Edge Deployment

```
┌──────────────┐
│Mobile Device │
│  ┌────────┐  │
│  │Tiny LLM│  │ (1-3B params, quantized)
│  └────────┘  │
└──────────────┘

Use cases:
- Offline chat
- Privacy-sensitive apps
- Low-latency needs
```

### 4. Hybrid Cloud-Edge

```
              ┌─────────┐
              │  User   │
              └────┬────┘
                   ↓
           ┌───────┴────────┐
           │                │
      ┌────▼────┐      ┌────▼─────┐
      │  Edge   │      │  Cloud   │
      │Small LLM│      │ Large LLM│
      │(Fast)   │      │(Powerful)│
      └─────────┘      └──────────┘

Routing:
Simple query → Edge
Complex query → Cloud
```

## Scaling and Load Balancing

### Horizontal Scaling

```
Single Instance:
┌──────────┐
│  1 GPU   │ → 100 requests/sec
└──────────┘

Scaled:
┌──────────┐
│  4 GPUs  │ → 400 requests/sec
└──────────┘

Linear scaling with GPU count
```

### Load Balancing Strategies

#### 1. Round Robin
```
Request 1 → GPU 1
Request 2 → GPU 2
Request 3 → GPU 3
Request 4 → GPU 1 (cycle)
```

#### 2. Least Connections
```
GPU 1: 5 active requests
GPU 2: 3 active requests ← Route here
GPU 3: 7 active requests

Send to least busy GPU
```

#### 3. Weighted Load Balancing
```
A100 (80GB): Weight 10
A10 (24GB): Weight 3

More requests → A100
Fewer requests → A10
```

### Auto-Scaling

```python
# Kubernetes HPA example
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: llm-inference
spec:
  scaleTargetRef:
    kind: Deployment
    name: llm-server
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: gpu-utilization
      target:
        type: Utilization
        averageUtilization: 70

# Scale up when GPU > 70%
# Scale down when GPU < 70%
```

## Monitoring and Observability

### Key Metrics

#### 1. Latency
```
Time to First Token (TTFT): 200ms
Time Per Output Token (TPOT): 50ms
Total Latency: TTFT + (N × TPOT)

Example (20 tokens):
Latency = 200ms + (20 × 50ms) = 1200ms

Monitor: p50, p95, p99 latencies
Alert: p95 > 2 seconds
```

#### 2. Throughput
```
Requests per Second (RPS): 50
Tokens per Second (TPS): 1000

Monitor over time:
Hour 1: 45 RPS ✓
Hour 2: 30 RPS ❌ (investigate)
```

#### 3. Resource Utilization
```
GPU Utilization: 85%
GPU Memory: 72GB / 80GB
CPU: 40%
Network: 100 Mbps

Alert if:
GPU Memory > 95% (OOM risk)
GPU Util < 30% (underutilized)
```

#### 4. Error Rate
```
Total Requests: 10,000
Errors: 50
Error Rate: 0.5%

Types:
- Timeout errors: 30
- OOM errors: 15
- Invalid input: 5

Alert if error rate > 1%
```

```python
import logging
```

## Cost Optimization

### Hardware Selection

```
Use Case → Hardware

Prototyping:
- Consumer GPU (RTX 4090) $1,600
- 24GB VRAM, good for 7B models

Production (Small Scale):
- A10 GPU $1-2/hour
- 24GB VRAM, inference optimized

Production (Large Scale):
- A100 GPU $2-3/hour
- 80GB VRAM, high throughput

Very Large Models (70B+):
- H100 GPU $4-6/hour
- 80GB HBM3, fastest performance
```

### Cost Calculation

```python
# Example: 7B model on A100

# Costs
GPU: $2.50/hour
Throughput: 100 requests/hour
Cost per request: $0.025

Monthly:
Requests: 1M
Cost: 1M × $0.025 = $25,000

vs OpenAI GPT-3.5:
1M requests: ~$2,000

Breakeven: ~80K requests/month
```

### Optimization Strategies

#### 1. Request Batching
```
Without batching:
100 requests/hour/GPU
Cost: $0.025/request

With batching (batch=8):
600 requests/hour/GPU
Cost: $0.004/request (6x cheaper!)
```

#### 2. Model Distillation
```
Large Model (70B):
- Quality: 95%
- Cost: $0.10/request

Distilled Model (7B):
- Quality: 90%
- Cost: $0.01/request (10x cheaper)

Use distilled for simple queries
```

#### 3. Caching
```
Cache common queries:

Cache hit rate: 30%
Cached response: $0 (instant)
Non-cached: $0.01

Effective cost: 0.7 × $0.01 = $0.007
Savings: 30%
```

#### 4. Auto-Scaling
```
Traffic pattern:
Peak (9AM-5PM): 4 GPUs needed
Off-peak (5PM-9AM): 1 GPU needed

Fixed: 4 GPUs × 24h = 96 GPU-hours/day
Auto-scaled: (4×8h) + (1×16h) = 48 GPU-hours/day

Savings: 50%
```

11. **Monitor p99 latency** for user experience

12. **Track GPU utilization** for cost optimization

13. **Auto-scaling** reduces costs by 30-50%

14. **Load balancing** ensures even distribution across GPUs

15. **Caching common queries** reduces cost and latency
