## Introduction to Fine-Tuning

### What is Fine-Tuning?

**Fine-tuning** is the process of adapting a pre-trained language model to perform better on specific tasks or domains by training it on task-specific data.

```
Pre-trained Model (General Purpose)
         ↓
    Fine-tuning on Task-Specific Data
         ↓
Specialized Model (Task-Specific)
```

### Pre-training vs Fine-tuning

| Aspect | Pre-training | Fine-tuning |
|--------|-------------|-------------|
| **Data** | Trillions of tokens | Thousands to millions of examples |
| **Goal** | Learn general language | Adapt to specific task |
| **Cost** | $1M-$10M | $100-$10K |
| **Time** | Weeks/months | Hours/days |
| **Frequency** | Once per model | Multiple times |
| **Hardware** | 100s-1000s of GPUs | 1-8 GPUs |

## When to Fine-Tune

### Fine-Tuning is Good For:

1. **Domain Adaptation**
```
Medical AI: Fine-tune on medical literature
Legal AI: Fine-tune on legal documents
Code AI: Fine-tune on code repositories
```

2. **Task Specialization**
```
Summarization: Fine-tune on summary pairs
Classification: Fine-tune on labeled data
QA: Fine-tune on question-answer pairs
```

3. **Style/Tone Adaptation**
```
Corporate tone: Professional, formal
Casual tone: Friendly, conversational
Technical tone: Precise, detailed
```

4. **Behavior Modification**
```
Safety: Reduce harmful outputs
Alignment: Match company values
Format: Always output JSON
```

### Use RAG Instead When:

```
❌ Don't Fine-Tune:
- Need up-to-date information
- Facts change frequently
- Need source attribution
- Knowledge base queries

✅ Use RAG:
- Document search
- Customer support (dynamic KB)
- News/current events
- Company knowledge base
```

### Use Prompting Instead When:

```
❌ Don't Fine-Tune:
- Simple task
- Limited examples
- Frequent task changes
- Quick experimentation

✅ Use Prompting:
- Few-shot learning works
- Task varies often
- No training data available
```

## Types of Fine-Tuning

### 1. Full Fine-Tuning

**Update all model parameters**

```python
# All layers trainable
for param in model.parameters():
    param.requires_grad = True

model.train()
```

**Characteristics**:
- Updates all ~7B-175B parameters
- Best performance
- Most expensive
- Requires significant GPU memory
- Risk of catastrophic forgetting

**Use Case**: When you have lots of data and compute

### 2. Feature Extraction (Frozen Base)

**Keep base model frozen, train only head**

```python
# Freeze base model
for param in model.base.parameters():
    param.requires_grad = False

# Train only classification head
for param in model.classifier.parameters():
    param.requires_grad = True
```

**Characteristics**:
- Fast and cheap
- Limited adaptation
- Good for simple classification

### 3. Partial Fine-Tuning

**Freeze some layers, train others**

```python
# Freeze early layers
for layer in model.layers[:20]:
    for param in layer.parameters():
        param.requires_grad = False

# Train later layers
for layer in model.layers[20:]:
    for param in layer.parameters():
        param.requires_grad = True
```

**Rationale**: Early layers learn general features, later layers are task-specific

## Parameter-Efficient Fine-Tuning (PEFT)

### What is PEFT?

Techniques that achieve comparable performance to full fine-tuning while training only a small fraction of parameters.

```
Full Fine-Tuning:
Train: 7B parameters
Memory: ~28GB
Time: Days

PEFT (LoRA):
Train: 4M parameters (0.06%)
Memory: ~16GB
Time: Hours
```

### Advantages of PEFT

1. **Lower Memory**: Fits on consumer GPUs
2. **Faster Training**: Fewer parameters to update
3. **Multiple Adapters**: Store multiple task-specific adapters
4. **No Catastrophic Forgetting**: Base model unchanged

### Popular PEFT Methods

#### 1. Adapter Layers

```
Original Transformer Layer:
Attention → FFN → Output

With Adapter:
Attention → Adapter → FFN → Adapter → Output
           (small MLP)        (small MLP)
```

#### 2. Prefix Tuning

```
Prepend trainable "prefix" vectors to each layer

Input: "Translate: Hello"
       ↓
Prefix (trainable) + Input → Layer 1
Prefix (trainable) + Hidden → Layer 2
...
```

#### 3. Prompt Tuning

```
Add trainable "soft prompts" to input

[Soft Prompt 1][Soft Prompt 2]...[Soft Prompt k] + "Classify: [text]"
└───────────────trainable──────────────────────┘
```

## LoRA and QLoRA

### LoRA (Low-Rank Adaptation)

**Key Idea**: Inject trainable low-rank matrices into transformer layers

```
Original Weight Update:
W_new = W_original + ΔW
(7B params to update)

LoRA:
W_new = W_original + B·A
Where A: [d × r], B: [r × d], r << d
(Only A and B are trainable - much smaller!)
```

### LoRA Mathematics

```
Original:  W ∈ ℝ^(d×d)  (e.g., 4096×4096)

LoRA adds: ΔW = B·A
           B ∈ ℝ^(d×r)  (e.g., 4096×8)
           A ∈ ℝ^(r×d)  (e.g., 8×4096)

Parameters:
Original: d² = 4096² = 16.7M
LoRA: 2·d·r = 2·4096·8 = 65K  (0.4%!)
```

### LoRA Implementation

```python
# Apply LoRA to linear layers
from peft import LoraConfig, get_peft_model

model = AutoModelForCausalLM.from_pretrained("base-model")

lora_config = LoraConfig(
    r=8,  # Rank
    lora_alpha=32,  # Scaling factor
    target_modules=["q_proj", "v_proj"],  # Which layers
    lora_dropout=0.1,
    bias="none",
    task_type="CAUSAL_LM"
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 4.2M || all params: 7B || trainable: 0.06%
```

### QLoRA (Quantized LoRA)

**Further optimization**: Quantize base model + LoRA

```
Base Model: Quantized to 4-bit
LoRA Adapters: Full precision (16-bit)

Benefits:
- Base model: 4-bit → 4x memory reduction
- Adapters: Still high quality (16-bit)
- Can fine-tune 65B model on single 48GB GPU!
```

```python
from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.float16
)

model = AutoModelForCausalLM.from_pretrained(
    "meta-llama/Llama-2-70b",
    quantization_config=bnb_config
)

# Apply LoRA on top
lora_model = get_peft_model(model, lora_config)
```

### LoRA Hyperparameters

```python
r (rank): 4, 8, 16, 32
- Lower r: Fewer params, less expressive
- Higher r: More params, more expressive
- Typical: 8-16

lora_alpha (scaling): 16, 32, 64
- Controls strength of LoRA update
- Typical: 2× rank

lora_dropout: 0.05-0.1
- Regularization

target_modules: Which layers to adapt
- Common: ["q_proj", "v_proj"]  (attention)
- More: ["q_proj", "k_proj", "v_proj", "o_proj"]
- All: Add feed-forward layers too
```

## Dataset Preparation

### Data Format

#### For Instruction Fine-Tuning

```json
[
  {
    "instruction": "Translate English to French",
    "input": "Hello, how are you?",
    "output": "Bonjour, comment allez-vous?"
  },
  {
    "instruction": "Summarize the text",
    "input": "[Long article text...]",
    "output": "This article discusses..."
  }
]
```

#### For Chat Fine-Tuning

```json
[
  {
    "messages": [
      {"role": "user", "content": "What is Python?"},
      {"role": "assistant", "content": "Python is a programming..."}
    ]
  }
]
```

#### For Completion Fine-Tuning

```json
[
  {"text": "Q: What is AI? A: AI stands for Artificial Intelligence..."},
  {"text": "Q: Explain transformers. A: Transformers are..."}
]
```

### Data Quality

```
✅ High Quality Data:
- Accurate labels
- Consistent format
- Diverse examples
- Representative of use case

❌ Low Quality Data:
- Noisy labels
- Inconsistent format
- Repetitive examples
- Not representative
```

### Data Quantity

```
Task Complexity → Data Needed

Simple Classification:
100-1,000 examples

Instruction Following:
1,000-10,000 examples

Domain Adaptation:
10,000-100,000 examples

General Purpose:
100,000+ examples
```

### Data Preprocessing

```python
def preprocess_data(example):
    # Format prompt
    prompt = f"Instruction: {example['instruction']}\nInput: {example['input']}\n"

    # Add output for training
    full_text = prompt + f"Output: {example['output']}"

    # Tokenize
    tokens = tokenizer(
        full_text,
        max_length=512,
        truncation=True,
        padding="max_length"
    )

    return tokens

dataset = dataset.map(preprocess_data)
```

## Training Process

### 1. Setup

```python
from transformers import (
    AutoModelForCausalLM,
    AutoTokenizer,
    TrainingArguments,
    Trainer
)
from peft import LoraConfig, get_peft_model

# Load base model
model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b")
tokenizer = AutoTokenizer.from_pretrained("meta-llama/Llama-2-7b")

# Configure LoRA
lora_config = LoraConfig(
    r=16,
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],
    lora_dropout=0.05,
    bias="none"
)

model = get_peft_model(model, lora_config)
```

### 2. Training Configuration

```python
training_args = TrainingArguments(
    output_dir="./results",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,  # Effective batch size: 16
    learning_rate=2e-4,
    fp16=True,  # Mixed precision
    logging_steps=10,
    save_steps=100,
    evaluation_strategy="steps",
    eval_steps=100,
    save_total_limit=2,
    load_best_model_at_end=True
)
```

### 3. Train

```python
trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset
)

trainer.train()
```

### 4. Save Model

```python
# Save LoRA adapters only
model.save_pretrained("./my-fine-tuned-model")
tokenizer.save_pretrained("./my-fine-tuned-model")

# Size: ~10-50MB (just LoRA weights!)
```

### 5. Load and Use

```python
from peft import PeftModel

# Load base model
base_model = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-2-7b")

# Load LoRA adapters
model = PeftModel.from_pretrained(base_model, "./my-fine-tuned-model")

# Generate
input_text = "Translate to French: Hello"
inputs = tokenizer(input_text, return_tensors="pt")
outputs = model.generate(**inputs)
print(tokenizer.decode(outputs[0]))
```

### Training Hyperparameters

```
Learning Rate:
- Full fine-tuning: 1e-5 to 5e-5
- LoRA: 1e-4 to 3e-4
- Higher for LoRA (adapters only)

Batch Size:
- As large as GPU memory allows
- Use gradient accumulation if needed
- Effective batch size: 16-64 typical

Epochs:
- 1-3 for large datasets
- 3-10 for small datasets
- Monitor for overfitting

Warmup Steps:
- 100-500 steps
- Gradual learning rate increase

Weight Decay:
- 0.01-0.1 for regularization
```

## Evaluation and Iteration

### 1. Training Metrics

```python
# Monitor during training
- Training Loss: Should decrease
- Validation Loss: Should decrease (watch for overfitting)
- Learning Rate: Follows schedule

Example:
Epoch 1: train_loss=2.5, val_loss=2.3
Epoch 2: train_loss=1.8, val_loss=1.9
Epoch 3: train_loss=1.4, val_loss=1.6

If val_loss stops improving: Early stop!
```

### 2. Holdout Evaluation

```python
# Test on held-out data
test_results = trainer.evaluate(test_dataset)

print(f"Test Loss: {test_results['eval_loss']}")
print(f"Test Perplexity: {np.exp(test_results['eval_loss'])}")
```

```python
test_prompts = [

for prompt in test_prompts:
```

### 4. A/B Testing

```
Serve both models:
- Baseline (original)
- Fine-tuned

Compare:
- User satisfaction
- Task success rate
- Response quality
```

### 5. Iteration

```
If performance insufficient:

1. Add more data
2. Improve data quality
3. Increase model size
4. Adjust hyperparameters
5. Try different PEFT method
6. Increase LoRA rank
```

## Key Takeaways

### Core Concepts

1. **Fine-tuning adapts pre-trained models** to specific tasks/domains

2. **PEFT methods** (LoRA, QLoRA) enable efficient fine-tuning with minimal parameters

3. **LoRA adds low-rank matrices** to model layers (0.1% of parameters)

4. **QLoRA quantizes base model** to 4-bit, enabling 65B+ model fine-tuning on single GPU

5. **Data quality matters more** than quantity for fine-tuning

### Practical Guidelines

6. **Use RAG** for dynamic knowledge, **fine-tuning** for behavior/style

7. **Start with LoRA** (r=8-16) for most tasks

8. **1K-10K examples** sufficient for instruction fine-tuning

9. **Monitor validation loss** to avoid overfitting

10. **Save LoRA adapters** (tiny files, easy to version)

### Best Practices

11. **Curate high-quality datasets** with diverse examples

12. **Use mixed precision (FP16)** to save memory

13. **Gradient accumulation** when batch size limited by memory

14. **Early stopping** prevents overfitting

15. **A/B test** fine-tuned model against baseline
